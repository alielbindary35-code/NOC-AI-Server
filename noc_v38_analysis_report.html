🛡️ Stability & Reliability

Fix critical bugs that could cause wrong results or silent failures in production

DO FIRST
🔴 CRITICAL

1.1 — Centralize Guardrail Logic (DRY Fix)

🐛 Problem
The same override rules for power+aging, warning/critical, and intent→category mapping are copied in 3 different nodes: INT_Operational Guardrails, CORE_Build Unified Context, ENGINE_Build SQL. A rule changed in one place but forgotten in another = silent production bugs.
✅ Solution
All intent+category correction logic lives only in INT_Operational Guardrails. The following nodes trust the output and never override again.
🔴 CRITICAL

1.2 — Pagination Error Handling

🐛 Problem
25 parallel API calls are fired for aging queries. If any single page fails (timeout, 500 error), the result is silently incomplete — the workflow never knows records are missing.
✅ Solution
Enable continueOnFail: true on the API node + track failed pages in ENGINE_Normalize. Better: use sequential pagination — fetch page 1 → read total → calculate pages needed → fetch only what's needed.
💡 Code Idea
// In ENGINE_Normalize: detect incomplete pagination const failedPages = pageInputs.filter(i => !i.json?.result); if (failedPages.length > 0) { // Include warning in reply warning = `⚠️ ${failedPages.length} pages failed — data may be incomplete`; }
🟠 HIGH

1.3 — Reliable Timezone Handling

🐛 Problem
Using toLocaleString() → new Date(string) to calculate Cairo timezone offset is unreliable. Different environments produce different string formats, causing NaN timestamps that corrupt the API time range.
✅ Solution
Use Intl.DateTimeFormat.formatToParts() which produces a structured object — never a locale-dependent string.
💡 Code
function getCairoMidnight() { const p = new Intl.DateTimeFormat('en-CA', { timeZone: 'Africa/Cairo', year:'numeric', month:'2-digit', day:'2-digit' }).formatToParts(new Date()); const d = Object.fromEntries(p.map(x => [x.type, x.value])); return new Date(`${d.year}-${d.month}-${d.day}T00:00:00`).getTime(); }

Maximize speed and improve the quality of responses users actually see

HIGH VALUE
🚀 PERF

2.1 — Fast Path Before LLM

🐛 Problem
qwen2.5:14b on Ollama takes 3–8 seconds per request. But then INT_Operational Guardrails overrides the LLM result anyway using simple keyword matching. For common queries like "How many site down today?" — the LLM adds zero value.
✅ Solution
Add a keyword-first classifier node after INT_Extract Message. If it matches with high confidence → skip LLM. If ambiguous → fallback to LLM. Expected latency improvement: 60–80% for common queries.
💡 Architecture
// New: ROUTE_Fast Path (before LLM) const FAST_RULES = [ { match: ['how many site down', 'كم موقع واقع'], intent: 'site_down_count' }, { match: ['power alarm', 'باور الارم'], intent: 'power_alarm_summary' }, { match: ['tx alarm', 'transport', 'microwave'], intent: 'tx_alarm_summary' }, ... ]; const hit = FAST_RULES.find(r => r.match.some(x => q.includes(x))); if (hit) return hit; // skip LLM → directly to guardrails
🟡 UX

2.2 — Better Empty-Result Messages

🐛 Problem
When the API returns 0 records, the reply shows numbers as 0 with no explanation. User doesn't know if there really are no incidents, or if the query was wrong.
✅ Solution
Detect zero-result responses in ENGINE_Format Reply and generate a smart message with the applied filters.
💡 Code
if (j.physical_site_count === 0) { reply = `✅ No ${intent} incidents found.\n` + `Applied: ${JSON.stringify(j.filters)} \n` + `Time: ${j.time_range.start_local} → ${j.time_range.end_local}`; }
📊 UX

2.3 — Dashboard Auto-Population

🐛 Problem
dashboard_data is only populated when the user writes "dashboard" in their question. Most users never know this keyword exists. Some intents have dashboard logic, others don't.
✅ Solution
Always populate dashboard_data for every non-free-chat response. Set response_type = 'mixed' always. Let the frontend decide whether to render the dashboard or not — it already has the logic for this.

3.1 — Intent Caching (5-minute TTL)

💡 Idea
Cache the full response for each unique question hash for 5 minutes. If the same query comes again → return instantly without LLM or API call. Implemented as an n8n Code node with a static Map or using Redis/KV store.
// Static in-memory cache (per n8n process) const cache = global.nocCache ??= new Map(); const key = hashQuestion(chatInput); const hit = cache.get(key); if (hit && Date.now() - hit.ts < 300_000) return [hit.result];
🔧 DX

3.2 — Free/Placeholder Unification Through Debug Node

💡 Idea
Currently FREE_Format and PLACEHOLDER_Format skip OUT_Build Debug Response and go directly to the webhook. Route all 3 paths through the Debug node for consistent debug output and a single exit point.
📋 DX

3.3 — Node Naming Cleanup

💡 Idea
Enforce the prefix convention consistently across all nodes:
Before → After "Basic LLM Chain" → "LLM_Classify Intent" "FREE_Format" → "OUT_Free Format" "PLACEHOLDER_Format" → "OUT_Placeholder Format" "ENGINE_Call Alarm API" → "API_Call Alarm API"
🚀 UX

3.4 — Streaming / Progressive Responses

💡 Idea
For aging queries (25 pages × API latency = up to 30s wait), send a confirmation message immediately, then stream progress updates, then the final result. Requires the frontend to support Server-Sent Events or WebSocket — a future enhancement.
