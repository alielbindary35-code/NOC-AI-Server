INT_Webhook

v38f-0001 · n8n-nodes-base.webhook
Input Gate
What it does

The entry point of the entire workflow. It listens on the URL path /webhook/chat-v38-foundation for HTTP POST requests. When the frontend sends a message, this node wakes up the workflow and passes the raw request body downstream.

Config: responseMode = responseNode — means it holds the HTTP connection open until the last node sends the reply (not immediate response). This is how the frontend gets the AI result back synchronously.

Input ↓ Output
// Frontend sends: POST /webhook/chat-v38-foundation { "message": "How many site down today?", "debug_mode": true } // Node output (passes the raw HTTP body): { body: { message: "How many site down today?", debug_mode: true } }
📥

LLM_Build Prompt

v38f-0003 · n8n-nodes-base.code
AI Prep
What it does

Builds the system prompt and user prompt that will be sent to the LLM. The system prompt is a detailed set of classification rules telling the AI how to interpret NOC questions. The user prompt is simply the user's question.

Key rules it teaches the LLM (examples):

IF "How many" + "site down" → intent = site_down_count, count_only = true
IF "power alarms" → intent = power_alarm_summary, category = POWER
IF "aging + down" → intent = down_aging_summary
IF "VIP down" → intent = vip_down_sites, isvip = Yes
IF region mentioned (Alex, Cairo...) → geo_type = region
IF site code mentioned (ALX3480...) → geo_type = site
Output
{ chatInput, debugMode, systemPrompt: "You are a telecom NOC...", userPrompt: "User question: ..." }
🤖

Basic LLM Chain + INT_Ollama Chat Model

v38f-0004/0005 · lmChatOllama + chainLlm
AI Inference
What it does

Sends the system + user prompt to qwen2.5:14b model running locally on Ollama (temperature=0 for deterministic output). The model reads the question and returns a JSON object with its classification.

⚠️ This step takes 3–8 seconds. The model is told to return ONLY raw JSON (no markdown).

Example LLM Output
// Input: "How many site down today in Alex?" // LLM returns (raw JSON string): { "mode": "incident", "intent": "site_down_count", "category": "SITE_DOWN", "geo_type": "region", "region_input": "Alex", "count_only": true, "sitedownflag": "Yes", "user_question": "How many site down today in Alex?" }
🔍

INT_Parse AI Output

v38f-0006 · code
Parser
What it does

Takes the raw text string from the LLM, parses it as JSON, and maps each field to a strongly-typed value. If parsing fails, all fields get safe empty defaults (never null).

Strategy: String(parsed.intent || '').trim() — every field is safely coerced to string.

Condition: Parse failure guard
TRY JSON.parse(chainOut)
→ SUCCESS: use the parsed object
→ FAIL: use empty {} — downstream guardrails handle the fallback
Output fields
// All 20+ fields normalised, e.g.: { intent: "site_down_count", // string, always lowercase category: "SITE_DOWN", // string, always uppercase geo_type: "region", region_input: "Alex", count_only: true, // boolean, from parsed or string 'true' debugMode: true, // carried from original request _aiRaw: "..." // raw LLM text for debug }
⚙️

INT_Operational Guardrails v3.9 OWNER

v38f-0007 · code
Safety Net
What it does

The single source of truth for intent+category correction. In v3.9, all the overriding logic lives HERE and nowhere else. It does 4 things:

1. Keyword-based hard overrides (keyword matching beats LLM for known patterns)

2. Fills missing intent when LLM returned empty/unknown

3. Enforces intent→category consistency

4. Sets request_profile (simple_query / site_incident_summary / free_chat / placeholder)

Key Conditions (Priority Order)
HARD OVERRIDE: "power" + "aging" in question → always = power_aging_summary + POWER
HARD OVERRIDE: "power" + "parked" → always = power_parked_summary + POWER
HARD OVERRIDE: "warning"/"critical" + "power" → power_aging_summary
IF intent is empty/unknown AND question has incident keywords → derive intent from keywords
→ "vip" in question → vip_down_sites + isvip=Yes
→ "tx"/"transport"/"microwave" → tx_alarm_summary
→ site code + "analyze" → site_incident_summary
→ "howmany"/"count" → site_down_count + count_only=true
IF greeting (hi/thanks/مرحبا) → free_chat
ELSE no incident keywords → placeholder
Examples
👤 "Power aging summary" hard override wins
intentpower_aging_summary
categoryPOWER
👤 "مرحبا" greeting path
modefree_chat
intentfree_chat
👤 "Show VIP sites down" keyword match
intentvip_down_sites
isvipYes
👤 "Traffic loss for ALX3480" future feature
intentplaceholder_request
modeplaceholder

CORE_Build Unified Context ✦ Timezone Fixed

v38f-0008 · code
Context Builder
What it does (v3.9)

Builds the canonical context object used by all downstream engine nodes. It does 3 things:

1. Geography resolution — maps region names to API format (Alex → "Alex", delta → "East Delta|West Delta")

2. Time range calculation — calculates today's start/end timestamps in Cairo time (fixed in v3.9)

3. Context packaging — combines filters, geography, time_range into one structured object

✅ v3.9 fix: Removed the duplicate guardrail overrides. Now trusts the guardrail output.

v3.9 — Timezone Fix Detail
// OLD (unreliable — environment-dependent string format): const nowStr = new Date().toLocaleString('en-US', { timeZone: 'Africa/Cairo' }); const cairoNow = new Date(nowStr); // ← string parsing = NaN risk! // NEW (correct — uses structured parts, not string parsing): const cairoParts = Object.fromEntries( new Intl.DateTimeFormat('en-CA', { timeZone: 'Africa/Cairo', ... }) .formatToParts(realNow) .filter(p => p.type !== 'literal') .map(p => [p.type, p.value]) ); // → { year:'2026', month:'04', day:'11', hour:'23', minute:'30', second:'45' } const cairoElapsedMs = (parseInt(cairoParts.hour)*3600 + ...) * 1000; let startUnix = realNow.getTime() - cairoElapsedMs; // = Cairo midnight UTC
Geography Mapping Examples
geo_type = region "alex" input
location_resolvedAlex
geo_type = region "delta" input
location_resolvedEast Delta|West Delta

ROUTE_Is Free Chat + ROUTE_Is Placeholder

v38f-0009/0010 · IF nodes
Decision Gate
What they do

Two sequential IF nodes that decide which path to take. They prevent unnecessary DB/API calls for simple or unsupported requests.

ROUTE_Is Free Chat — Conditions (OR)
IF request_profile == "free_chat"
OR mode == "free_chat"
OR intent == "free_chat"
→ TRUE: go to FREE_Format (instant reply, no API)
→ FALSE: check ROUTE_Is Placeholder
ROUTE_Is Placeholder — Conditions (OR)
IF request_profile == "placeholder"
OR mode == "placeholder"
OR intent == "placeholder_request"
→ TRUE: go to PLACEHOLDER_Format (explain & ask for more info)
→ FALSE: go to ENGINE_Build SQL (full incident path)
Routing Summary
👤 "Hi / Thanks / مرحبا"
path→ FREE_Format
👤 "Traffic loss prediction"
path→ PLACEHOLDER_Format
👤 "How many site down today?"
path→ ENGINE path ▶
👤 "Analyze site ALX3480"
path→ ENGINE path ▶

FREE_Format + PLACEHOLDER_Format

v38f-0011/0012 · code
Fast Reply
What they do

FREE_Format: Returns a friendly greeting/acknowledgment instantly. No DB/API involved. Detects "thanks" specifically for a warmer reply.

PLACEHOLDER_Format: Returns a helpful message explaining what's needed (e.g. "Please provide a site code") or that the capability is coming soon. Includes the debug object.

FREE_Format condition
IF question matches /thanks|thank you|شكرا/i → "😊 You are welcome..."
ELSE → "👋 Hello! I am the NOC AI Assistant..."
Both output the standard contract
{ version: "v1", response_type: "text", intent: "free_chat", reply: "...", summary: {}, charts: {}, tables: {}, debug: { ... } // full debug object }

ENGINE_Build SQL ✦ Simplified

v38f-0013 · code
SQL Builder
What it does

Builds a SQL query to fetch the alarm names associated with the current intent's category from the DB mapping table ai_alarm_mappings. These alarm names are then used to filter the API results.

✅ v3.9 fix: Removed the redundant intent/category override block. Now trusts guardrails completely.

Conditions
IF category is SITE_DOWN, POWER, TX, or CELL_DOWN → build SQL WHERE clause
→ SELECT alarm_name FROM ai_alarm_mappings WHERE category = '...'
IF request_profile = site_incident_summary OR intent = site_active_alarms
→ SELECT '__DIRECT__' (bypass alarm mapping, query site directly)
SQL Examples
intent = site_down_count (SITE_DOWN)
SELECT alarm_name FROM ai_alarm_mappings WHERE category = 'SITE_DOWN' ORDER BY alarm_name;
intent = site_active_alarms (__DIRECT__)
SELECT '__DIRECT__' AS alarm_name

v38f-0014 · postgres
Database
What it does

Executes the SQL query built by the previous node against the PostgreSQL database. Returns a list of alarm names that belong to the requested category. These alarm names are later used to filter API results to the right alarm types.

Output Example
// For SITE_DOWN category: [ { alarm_name: "Site Down" }, { alarm_name: "Site Unreachable" }, { alarm_name: "Link Failure" }, // ... ] // For __DIRECT__ mode: [{ alarm_name: "__DIRECT__" }] // signals: no alarm-name filter

ENGINE_Build API Payload

v38f-0015 · code
Payload Builder
What it does

Constructs the API payload (headers) to send to the alarm API. Only adds a field if it has a value (no empty filters sent to API). Special handling for aging/parked intents (no severity filter).

Key Conditions
IF isDirectMode → do NOT send alarmname filter (query everything for the site)
IF geo_type = region → add location header
IF geo_type = office → add city header
IF isAgingIntent → never send standardalarmseverity (aging is time-based, not severity)
ELSE → add severity if Critical/Major/Minor/Warning in question
Example payload for "site down in Alex"
{ start: 0, limit: 1000, type: 1, starttime: 1712793600000, // today Cairo midnight UTC endtime: 1712879999999, // today Cairo 23:59:59 UTC alarmname: "Site Down|Site Unreachable|Link Failure|...", location: "Alex", // from geo resolution sitedownflag: "Yes" }
📄

ENGINE_Prepare Page Requests

v38f-0015b · code
Pagination
What it does

Splits the request into multiple paginated API calls by generating N copies of the payload with different start offsets. All pages fire in parallel to the API.

Page count logic
IF isAging (aging/parked intent) → maxPages = 25 (large dataset expected)
ELSE IF isBroad (site_down_count, power, tx...) → maxPages = 10
ELSE (specific site query) → maxPages = 1
Output example (3 pages, limit=1000)
// 3 separate items sent to ENGINE_Call_Alarm_API in parallel: [ { api_payload: { start: 0, limit: 1000 } }, { api_payload: { start: 1000, limit: 1000 } }, { api_payload: { start: 2000, limit: 1000 } } ]
📡

ENGINE_Call Alarm API

v38f-0016 · httpRequest
HTTP Client
What it does

Makes the actual HTTP GET request to the alarm API (ADC integration API). All filters are sent as request headers. Uses Basic Auth and allows self-signed SSL certs (internal server).

⚠️ All pages from the previous node are called in parallel, each with a different start offset.

API Endpoint
GET https://10.80.174.137:31943/adc-intg/api/rest/v1/ custom_api/query_alarms/query_alarm Headers: start, limit, type, starttime, endtime, alarmname, location, city, sitecode, vendor, networktype, isvip, sitedownflag, ...

ENGINE_Normalize And Dedup ✦ Error Tracking

v38f-0017 · code
Data Engine
What it does (largest node)

Collects all paginated API responses and processes them into clean, analysis-ready data. Does 6 things:

1. Page collection — merges all page responses into one allRecords array

2. Failed page detection (v3.9 new) — tracks which pages returned nothing

3. Post-filtering — applies vendor/tech/sitecode filters that couldn't go in API headers

4. Deduplication — gets latest alarm per site (one row per site, not 50 alarms)

5. Breakdowns — tallies by office, region, technology, alarm name

6. Aging analysis — computes how long sites have been down (<2h, 2-4h, 4-12h, >12h)

v3.9 — Pagination Error Tracking
// NEW in v3.9: detect pages that returned no data const failedPageIndices = pageInputs .map((item, i) => ({ i, raw: item.json || {} })) .filter(({ raw }) => !raw.result && !raw.data && !raw.records ...) .map(({ i }) => i); // e.g. [2, 7] = pages 2 and 7 failed const paginationWarning = failedPageIndices.length > 0 ? 'Data may be incomplete: N pages failed.' : null;
Key Dedup Logic
getLatestPerSite(rows) → for each unique sitecode, keep the record with the most recent timestamp
Result: physical_site_count = unique sites affected (not alarm count)
filterRowsByAge(rows, 24) → keep only rows older than 24h → parked sites
Output Example
{ physical_site_count: 47, // unique sites down logical_count: 183, // raw row count (multiple alarms per site) breakdowns: { by_office: [{ name: 'ALEX1', count: 8 }, ...], by_region: [{ name: 'Alex', count: 12 }, ...] }, aging_analysis: { down: { '<2h': 5, '2_4h': 12, '4_12h': 18, '>12h': 12 } }, pagination: { failed_pages: [], // v3.9 new data_complete: true // v3.9 new } }

ENGINE_Format Reply

v38f-0018 · code
Formatter
What it does

The biggest switch statement in the flow. Based on the intent, it builds the human-readable reply text, summary object, charts, and tables for the frontend dashboard.

Intent → Format mapping
site_down_count → KPI summary + office/region/tech charts + top sites table
power_alarm_summary → power impact metrics + alarm type table
down_aging_summary → aging distribution chart (<2h, 2-4h, 4-12h, >12h)
vip_down_sites → VIP-only list with office breakdown
site_incident_summary → 7-section full incident report for one site
site_active_alarms → alarm list table for one site
down/power_parked → parked (>24h) breakdown with office chart

OUT_Respond to Webhook

v38f-0020 · respondToWebhook
Response Gate
What it does

Closes the HTTP connection opened by the Webhook node and sends the final JSON response to the frontend. Adds CORS headers so the browser can receive the response from any origin.

Final Response Schema
{ version: "v1", response_type: "mixed", // or "text" / "dashboard" intent: "site_down_count", category: "SITE_DOWN", reply: "═══ Site Down Summary ═══\n...", total: 47, summary: { total_down_sites: 47, vip_sites: 3 }, charts: { by_technology: { ... }, by_region: { ... } }, tables: { by_office: { ... }, top_sites: { ... } }, meta: { time_range: { ... }, generated_at: "..." }, debug: { /* 12 sections */ }, dashboard_data: null // or populated if output_view=dashboard }
